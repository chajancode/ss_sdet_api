# Архитектурные решения

Решения по проекту и почему они приняты именно так.

## 1. Слоистая архитектура и разделение ответственности

**Было:** логика чтения из БД была размазана - DAO возвращал сырые кортежи, отдельный конвертер (`tuple_to_post_model`) превращал их в модели, а сервис знал и про кортежи, и про SQL-детали.

**Стало:** четыре слоя с чёткими ролями:
- **API-клиент** - только транспорт: запрос, статус, десериализация тела в pydantic-модель
- **Service** - сценарий: вызвать API и сверить результат с БД
- **Repository** - переводчик: принимает/отдаёт доменные модели, маппинг строки БД в модель внутри себя
- **DAO** - инкапсулирует SQL, возвращает сырьё

Поток: `test -> service -> api_client -> API`, а для сверки `service -> repository -> dao -> MySQL`. Смена схемы БД правит только DAO, остальные слои не трогаются.

## 2. Repository поверх DAO

DAO и Repository - не дубли, а два уровня: DAO говорит на языке БД (SQL, строки), Repository - на языке домена (`DBPostData`). Маппинг живёт в репозитории (`_row_to_model`), поэтому сервис работает с объектами, а не с `row[0][0]`.

## 3. Сверка ответов API с реальным состоянием БД

API может вернуть 200 и при этом не записать (или не удалить) данные так, как ожидалось. Поэтому сервис после операции читает соответствующую запись из БД через репозиторий и отдаёт её в тест вместе с ответом API - тест проверяет обе стороны: и тело ответа, и факт в базе.

**Урок:** при рефакторинге всплыл баг в проверке удаления - метод удалял запись по переданному `id`, а сверял в БД по `_last_created_id` (другой записи). Сверять надо тот же id, который меняли, иначе тест "зелёный" на несогласованных данных.

## 4. Типизированный HTTP-клиент со своей моделью ошибки для каждого API

**Было:** клиент был жёстко завязан на `WordPressError` - любой не-WordPress ответ пытался разобрать чужую ошибку как ошибку WordPress.

**Стало:** `FullAPIResponse[M, E]` - дженерик над телом успеха и телом ошибки; модель ошибки задаётся при создании клиента (`WordPressError` для WP-сервисов, `YandexApiError` для Яндекса) и переопределяется для каждого отдельного запроса. Клиент стал транспортом, не знающим про конкретный бэкенд.

## 5. Единый механизм доступа к БД

**Было:** два класса - `Database` (новое соединение на каждый запрос) и `DatabaseSession` (одно соединение). Дублирование с разной семантикой.

**Стало:** один `DatabaseSession`. Он умеет и `select`, и `execute`, возвращающий `lastrowid` (из `Database` id новой записи было не достать). `autocommit=True` тут важен: соединение живёт весь прогон, и без автокоммита чтения ловили бы устаревший снимок (REPEATABLE READ) и не видели изменений, сделанных через API. Компромисс: одно долгоживущее соединение быстрее, но на очень долгом прогоне может отвалиться по таймауту.

## 6. Тестируемость: подмена репозитория

Сервис принимает репозиторий как зависимость, поэтому в юнит-тесте его можно заменить фейком, который отдаёт готовый `DBPostData`. Логика сервиса проверяется без поднятой MySQL. Это и есть практический выигрыш слоя репозитория - тесты своего кода не привязаны к инфраструктуре.

## 7. Валидация структуры по JSON Schema

Pydantic-модели проверяют поля, которые мы явно описали. Для ответа со списком файлов Яндекс.Диска дополнительно валидируется вся структура по JSON Schema - это фиксирует контракт формата ответа целиком, а не только те поля, что нужны конкретному тесту.
